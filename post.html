Go 并发编程

深入理解 Go 语言内存模型

Go 的内存模型对于编写正确的并发程序至关重要。本文从 Happens-Before 关系入手， 分析各种同步原语的内存语义。

Go 语言以其简洁的语法和强大的并发支持而闻名。然而，要编写正确的并发程序， 理解其内存模型是必不可少的。本文将深入探讨 Go 内存模型的核心概念， 帮助读者建立系统的理解。

什么是内存模型

内存模型定义了程序中各个内存操作的执行顺序，以及在多核处理器上这些操作如何被观察到。 简单来说，它回答了这个问题：当一个 goroutine 写入一个变量时， 另一个 goroutine 何时能看到这个写入？

"If you must read the rest of this document to understand the behavior of your program, you are being too clever. Don't be clever." —— Go Memory Model

Happens-Before 关系

Go 内存模型使用 Happens-Before 关系来定义内存操作的顺序。 如果事件 A happens-before 事件 B，那么 A 的效果对 B 可见。

在单个 goroutine 中，程序的顺序就是 happens-before 关系：

var a, b int

func f() {
    a = 1  // 写入 a
    b = 2  // 写入 b，happens-after 写入 a
}

同步原语

Channel 通信

Channel 是 Go 中最主要的同步机制。对于同一个 channel：

  • 发送操作 happens-before 对应的接收操作完成
  • 关闭 channel happens-before 接收到零值
  • 从无缓冲 channel 接收 happens-before 发送完成
var done = make(chan bool)
var msg string

func main() {
    go func() {
        msg = "hello, world"
        done <- true
    }()
    <-done  // 接收 happens-after 发送
    println(msg)  // 保证看到 "hello, world"
}

Mutex

对于 sync.Mutex 或 sync.RWMutex：

  • 第 n 次 Unlock happens-before 第 n+1 次 Lock

WaitGroup

WaitGroup 的 Done 操作 happens-before Wait 返回。 这保证了所有 goroutine 完成的工作对等待者可见。

常见的错误模式

以下是一些常见的并发错误：

// 错误：没有同步机制
var flag bool
var data int

func worker() {
    data = 100
    flag = true
}

func main() {
    go worker()
    for !flag {  // 可能永远看不到 flag 的变化
    }
    println(data)  // 可能看到 0
}

正确的做法是使用 channel 或 mutex 进行同步：

// 正确：使用 channel 同步
var data int
done := make(chan struct{})

func worker() {
    data = 100
    close(done)
}

func main() {
    go worker()
    <-done  // 等待同步信号
    println(data)  // 保证看到 100
}

总结

理解 Go 的内存模型对于编写正确的并发程序至关重要。关键要点：

  1. 单个 goroutine 内的操作按程序顺序执行
  2. 使用 channel、mutex 等同步原语建立 happens-before 关系
  3. 避免依赖非同步的内存可见性
  4. 保持简单，优先使用高层抽象
